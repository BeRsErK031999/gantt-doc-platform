# AGENTS.md

## 🎯 Основная цель

Все действия в проекте должны соответствовать архитектуре `gantt-doc-platform` и принципам типобезопасности, предсказуемости, минимальной сложности и поэтапного развития.

Проект создаётся как верхнеуровневая document platform для работы с документами и связанными сценариями поверх будущих document flows:
- frontend (`apps/web`)
- backend (`apps/api`)
- общие типы (`packages/shared-types`)
- документация и архитектурные решения (`docs/*`)
- служебные скрипты (`scripts/*`)

На ранних этапах проект не должен притворяться готовой production-платформой. Сначала строится чистый foundation, затем — минимальный workspace/toolchain, затем — управляемый scaffold приложений, и только потом — доменная и прикладная функциональность.

При генерации или изменении кода необходимо:

1. Понимать цель текущего шага и не выходить за его рамки.
2. Анализировать существующую структуру проекта перед изменениями.
3. Вносить минимально необходимые, типобезопасные и масштабируемые правки.
4. Не придумывать архитектуру сверх задачи.
5. Обеспечивать прохождение проверок проекта, если они уже настроены.

Если в проекте уже настроены проверки, обязательно учитывать прохождение:

```bash
yarn type-check
yarn lint
```

Если проверки ещё не настроены, код всё равно должен быть написан так, как будто эти проверки будут добавлены сразу после текущего шага.

## ✔️ Обязательные правила

### 1. Работа только в рамках текущей задачи

- Выполнять только то, что явно запрошено.
- Не добавлять новые сервисы, библиотеки, директории, конфиги и архитектурные слои без прямой необходимости.
- Не переходить к следующему этапу, если текущий не завершён и не проверен.
- Если задача касается только структуры файлов — не устанавливать зависимости.
- Если задача касается только foundation — не создавать заранее `src`, доменные слои и runtime-код.
- Если задача касается только конфигурации — не писать бизнес-логику.
- Если задача касается только одного приложения (`apps/api` или `apps/web`) — не изменять второе без необходимости.

### 2. Работа с TypeScript

- Все типы импортировать **только через `import type`**.
  ```ts
  import type { DocumentDto } from "../model/types"
  ```
- Не создавать дублирующие типы, если они уже существуют.
- Использовать строгую типизацию.
- Не допускать:
  - `any`
  - `as any`
  - `as unknown as`
  - неявных опасных приведений типов
- Если тип не определён — сначала создать корректный тип, а не обходить проблему приведением.

### 3. Архитектура monorepo

Структура проекта должна оставаться понятной и устойчивой:

- `apps/web` — frontend
- `apps/api` — backend
- `packages/shared-types` — общие типы между приложениями
- `docs/*` — документация, архитектурные решения, notes, ADR
- `scripts/*` — локальные служебные скрипты проекта

Обязательные правила:
- не смешивать frontend и backend код между собой;
- не размещать runtime-код в `docs`;
- не размещать бизнес-логику в `scripts`;
- общие сущности выносить в `packages` только если они реально используются в нескольких местах;
- не выносить код в `packages` преждевременно, если он нужен только одному приложению;
- не создавать дополнительные пакеты конфигурации заранее, пока не доказана их необходимость.

### 4. Правила foundation-этапа

Пока проект находится на раннем foundation-этапе, запрещено без отдельного прямого запроса добавлять:

- базы данных;
- ORM;
- очереди;
- кэш-хранилища;
- auth;
- upload;
- AI-интеграции;
- DOCX parsing;
- интеграцию с внешними сервисами;
- production-ready инфраструктуру;
- сложные доменные слои, если ещё не завершён базовый scaffold.

Foundation-этап должен оставаться тонким:
- минимум файлов;
- минимум решений "на будущее";
- никакой ложной готовности API/Web;
- никакого полуготового монолита из пустых заготовок.

### 5. Правила backend (`apps/api`)

Когда backend будет вводиться, он должен оставаться предсказуемым и простым.

- Маршруты, типы, прикладная логика, интеграции и инфраструктура должны быть разделены по ответственности.
- Нельзя смешивать:
  - HTTP-слой,
  - доменную модель,
  - инфраструктуру,
  - конфигурацию,
  - интеграции с внешними сервисами.
- Работа с `process.env` должна быть централизованной, а не разбросанной по коду.
- Нельзя хардкодить URL внешних сервисов прямо в route-handlers.
- Нельзя добавлять базу данных, очереди, Redis, брокеры сообщений и тяжёлую инфраструктуру без прямого запроса.

### 6. Правила frontend (`apps/web`)

Когда frontend будет вводиться, UI должен оставаться простым, чистым и расширяемым.

- Не смешивать:
  - API-вызовы,
  - рендеринг,
  - преобразование данных,
  - конфигурацию.
- Не помещать тяжёлую логику преобразования данных прямо в компоненты.
- Компоненты должны иметь понятные `props`.
- Переиспользуемые UI-части выносить только тогда, когда это реально уменьшает дублирование.
- Не строить сложную дизайн-систему до появления реальной необходимости.
- На раннем этапе предпочтительны простые и понятные компоненты вместо переусложнённой абстракции.

### 7. Работа с document platform

Проект строится вокруг документов как сущностей. Поэтому:

- документ должен рассматриваться как доменная сущность, а не просто как файл на диске;
- нельзя заранее зашивать платформу под один-единственный формат;
- нельзя смешивать storage concerns, document metadata, derived artifacts и будущие operation flows в одном месте;
- если вводятся document types, statuses, DTO или контракты, они должны быть ясными и минимальными;
- нельзя придумывать сложную document pipeline-архитектуру до появления реальных сценариев.

### 8. Работа с Docker и инфраструктурой

- На раннем этапе используется только минимально необходимая инфраструктура.
- Не добавлять:
  - nginx
  - traefik
  - postgres
  - redis
  - message brokers
  - minio
  - k8s manifests
  - CI/CD pipelines
  - production secrets
  без прямой задачи.
- `docker-compose.yml` должен оставаться простым и читаемым.
- Если compose пока не нужен, допускается временная заглушка с честным описанием.
- Не заменять простое решение сложным без причины.

### 9. Проверка изменений

Перед завершением любого изменения необходимо:

1. Проверить, какие файлы реально были изменены.
2. Убедиться, что изменения соответствуют текущей задаче.
3. Не трогать посторонние файлы.
4. Убедиться, что проект не усложнён без необходимости.
5. Если доступны команды проверки — учитывать их прохождение:

```bash
yarn type-check
yarn lint
```

Если возникают ошибки TypeScript или линтера:

1. Определить причину.
2. Исправить реализацию корректно, а не обходным путём.
3. Не использовать небезопасные приведения типов ради "быстрого зелёного результата".
4. Повторно проверить результат.

### 10. Формат кода и имена

- Использовать `ts`, `tsx`, не `js`, `jsx`, если только шаг явно не требует иного.
- Компоненты: `PascalCase`
  - `DocumentsPage`
  - `DocumentCard`
- Переменные и функции: `camelCase`
  - `listDocuments`
  - `buildDocumentDto`
- Константы верхнего уровня: понятные и осмысленные имена.
- Не использовать имена:
  - `data1`
  - `temp`
  - `val`
  - `test123`
  - `handler2`
- Все сущности должны иметь ясные имена, отражающие назначение.

### 11. Работа с зависимостями

- Не устанавливать зависимости без необходимости.
- Перед добавлением новой библиотеки оценить:
  1. можно ли решить задачу без неё;
  2. не дублирует ли она уже существующий инструмент;
  3. не добавляет ли она лишнюю сложность.
- Не добавлять тяжёлые библиотеки "на будущее".
- Не менять пакетный менеджер без прямого указания.
- Если пакетный менеджер уже выбран в проекте — придерживаться его во всех дальнейших шагах.

### 12. Работа с Git

- Не делать коммиты, теги, push, rebase, merge или создание веток без явной команды.
- Не переписывать историю git.
- Не удалять файлы без объяснения причины.
- Перед изменениями полезно показать список текущих файлов, если есть риск затронуть лишнее.
- Если меняются существующие файлы — делать это аккуратно и минимально.

### 13. Работа с документацией

- README и документация должны отражать реальное состояние проекта.
- Не писать в документации то, чего ещё нет в коде.
- Не создавать видимость готового production-решения, если проект находится на раннем этапе.
- Все описания должны быть конкретными и проверяемыми.

## 📄 Итог

Все изменения в проекте должны быть:

- типобезопасными,
- минимально необходимыми,
- понятными,
- масштабируемыми,
- проверяемыми,
- соответствующими текущему этапу разработки.

Главное правило:
**не усложнять проект раньше времени и не выходить за рамки задачи.**

Если в проекте настроены проверки, итоговый код должен проходить:

```bash
yarn type-check
yarn lint
```

Изменения, нарушающие эти требования, недопустимы.

Никогда не отправляй в изменения служебные локальные артефакты пакетного менеджера и среды разработки, если они не относятся к сути задачи. Работай с ними локально только при необходимости, но не включай их в итоговые изменения без явной причины.
